# Ankiカード作成スキル設計

## 目的

学習対象と既存のAnki構成を調べ、答えやすく続けやすい暗記カードを作るスキルを追加する。デッキ名、ノートタイプ、フィールド名、タグ、カード形式は固定しない。その案件で使われている構成を優先し、不明な契約だけ利用者へ確認する。

成果物はAnkiのテキストインポート形式TSVと、同じ正規データから生成したMarkdownプレビューとする。新規カードの作成と、既存カードを学習履歴を保って更新する作業を区別する。

## 配置

スキルは配布対象として次へ置く。

```text
dist/.apm/skills/create-anki-cards/
├── SKILL.md
├── references/
│   └── card-design.md
└── tools/
    └── anki-cards.ts
```

`dist/apm.yml` のローカル依存へ `./.apm/skills/create-anki-cards` をアルファベット順で追加する。`apm.lock.yaml` と配布先ごとの生成物は手で変更せず、既存の `agents:build` taskで再生成する。

## 作業プロジェクトの構成

スキルを使う案件では、原則として次のファイルを扱う。

```text
anki.json
ANKI_PLAN.md
cards.tsv
cards.preview.md
```

`anki.json` を唯一の正規データとし、出力契約とカード本文をまとめて保持する。TSVとプレビューは生成物であり、手編集しない。

`ANKI_PLAN.md` は複数回に分けて続ける案件だけで使う。到達目標、判断理由、未解決事項、弾ごとの進捗を記録する。デッキ名やフィールド順など、CLIが解釈する契約を重複して持たせない。

## `anki.json`

最上位に `version`、`contract`、`cards` を持たせる。

`contract` は少なくとも次を表現する。

- 出力ファイル名
- 新規作成または既存更新
- デッキ
- ノートタイプ
- Ankiへ渡すフィールドの名前と順序
- 各フィールドの役割
- HTMLの使用有無
- タグ規約
- 既存更新時の安定識別子

フィールドの役割は `question`、`answer`、`reference`、`media`、`id`、`other` から指定できるようにする。同じ役割を複数フィールドへ割り当てることは許す。役割は品質監査に使い、Anki上のフィールド名を制限しない。

各カードは次を持つ。

- Ankiへ渡さないレビューID
- フィールド名をキーとする値
- タグ
- 一次資料
- 必要な場合だけ、判断理由や注意事項

レビューIDは案件内で不変かつ一意にする。新規カードのAnki GUIDとしては使わない。

## 識別方法

Anki GUIDはAnkiが生成する内部識別子として扱う。新規作成時に独自GUIDを生成しない。

新規カードは、既存ノートタイプに管理ID用フィールドがある場合だけ、通常フィールドへ安定IDを出力できる。管理ID用フィールドがない場合、レビューIDは `anki.json` にだけ保存する。

既存カードを更新する場合は、AnkiからGUIDを含めて書き出したTSVを入力にする。`contract` で指定した安定フィールドまたはURLを使い、`anki.json` のカードと書き出し行を一対一で照合する。重複、不足、余剰があれば更新TSVを生成しない。照合できたAnki由来のGUIDだけを更新TSVへ出力する。

Ankiの実データベースは編集しない。インポートはAnkiの画面から行う。

## スキルの流れ

1. 継続案件では `anki.json` と `ANKI_PLAN.md` を読む。
2. プロジェクト規約、既存TSV、Ankiのテキスト書き出しを確認する。
3. デッキ、ノートタイプ、フィールド、カード形式、新規作成か既存更新かを確定する。
4. 到達目標、対象範囲、苦手領域、1弾の枚数を確認する。既定は20〜30枚とするが、利用者の指定を優先する。
5. コード、設定、仕様書、公式文書などの一次資料を読む。
6. 学習対象の前提関係を整理し、必要な基礎カードを先に作る。
7. カードを `anki.json` へ追加する。
8. CLIで検査し、TSVとプレビューを生成する。
9. 利用者がプレビューを読み、レビューIDで修正を指示する。
10. 修正後に再生成し、インポート方法と確認事項を提示する。

語彙を「一般基礎、プラットフォーム基礎、対象固有」の三階層へ必ず分けることはしない。技術領域で有効な分類例としてリファレンスに残し、対象に合う前提関係をその都度決める。

Basic一方向、Cloze、逆カードも固定しない。既存契約がなければ、一つの想起対象を問う一方向カードを推奨する。複数カードを生成するノートタイプは、復習量への影響を利用者が了承してから使う。

## CLI

Bunで実行する単一のCLIを用意する。

```text
bun tools/anki-cards.ts check anki.json
bun tools/anki-cards.ts build anki.json
bun tools/anki-cards.ts build anki.json --anki-export exported.tsv
```

`check` は次を検査する。

- JSONと契約の構造
- レビューIDの空欄と重複
- 契約フィールドとカードフィールドの一致
- 必須フィールドの空欄
- タグの重複と許可範囲
- 一次資料の有無
- NULなどAnkiのテキストインポートへ渡せない制御文字
- 複数回答や二重質問を疑う表現
- 答えが長すぎる可能性

構造や一意性の違反はエラーとする。文章品質に関する検出は誤検知があるため警告とし、警告ごとに人が判断した結果を記録できるようにする。

`build` は内部で `check` と同じ検査を行う。エラーがなければ、同じ `anki.json` からTSVとMarkdownプレビューを生成する。生成途中の失敗で片方だけ更新されないよう、一時ファイルへ書いてから置き換える。既存の生成物以外は上書きしない。

`--anki-export` を指定した場合は既存更新として扱い、完全照合に成功した場合だけGUID列付きTSVを生成する。

## カード品質

カードは一つの想起対象へ絞り、表面から5〜15秒程度で答えられる粒度を目安にする。列挙を丸ごと問わず、一項目ずつ分けるか、既知の項目を表面へ与えて残り一つを問う。

表面には対象、前提、答えの種類を書き、指示語だけで終わらせない。裏面は想起対象を先に置き、補足はフィールド契約が許す場所へ分ける。特定の「裏面」フィールドやHTML構造は前提にしない。

カードで使う用語に未学習の前提がある場合は、前提カードを先に作る。定義カードが揃った領域には、症状や状況から判断を引き出す応用カードを追加できる。

事実は一次資料で確認し、カードごとに根拠を記録する。変わりやすい数値は、暗記すること自体が到達目標に必要で、更新方法まで決まっている場合だけカード化する。

詳細な文面規約と監査例は `references/card-design.md` に置き、`SKILL.md` は実行手順に絞る。

## 検証

CLIにはBun標準テストを追加し、少なくとも次を確認する。

- 任意の日本語フィールド名と順序でTSVを生成できる
- HTMLの有効・無効がヘッダーへ反映される
- タブ、改行、引用符を含む値を正しくTSVへ書ける
- レビューID、タグ、識別子の重複を拒否する
- カードの不足フィールドと未知フィールドを拒否する
- 同じ正規データからTSVとプレビューを再生成できる
- 新規作成TSVへGUIDを出力しない
- Anki書き出しとの完全照合に成功した場合だけ更新TSVへGUIDを出力する
- 生成途中で失敗しても既存成果物を壊さない

スキルフォルダはskill-creatorの検証スクリプトでも検査する。最後にdotfilesの `agents:build` と `dev:test` を実行し、APMへの登録、Claude/Codex向け生成物、既存設定の回帰を確認する。

TSVでは、Ankiの公式なテキスト形式に従ってタブ、改行、引用符を二重引用符で囲み、内部の引用符を二重化する。NULなど、テキストインポートで安全に表現できない制御文字だけを拒否する。
